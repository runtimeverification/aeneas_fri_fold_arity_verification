# Aeneas FRI Fold-Arity Verification

Proof-of-concept for verifying Plonky3 FRI cryptographic code using
**Aeneas** (MIR-based, pure functional extraction) as an alternative to
**Hax** (annotation-driven, monadic extraction).

Reference Hax project: [p3-hax-lean-fri-pipeline](https://github.com/runtimeverification/p3-hax-lean-fri-pipeline)

## Pipeline

```
Rust source  ──Charon──▸  LLBC  ──Aeneas──▸  Lean 4  ──proofs──▸  verified properties
fold_arity/src/lib.rs    .llbc             FoldArity.lean        BasicProofs.lean
```

## Hax vs Aeneas Comparison

| Dimension | Hax | Aeneas |
|-----------|-----|--------|
| Extraction input | Rust HIR + annotations | Rust MIR (via Charon LLBC) |
| Monad | `RustM` (custom inductive) | `Result` (ok / fail / div) |
| Option type | `Core_models.Option.Option` (reversed ctors) | Standard Lean `Option` |
| Conditionals | `Bool.casesOn` — `split_ifs` silently fails | Standard `if/then/else` — `split_ifs` works |
| Checked arithmetic | Requires custom `-?` lemmas | Built-in via `UScalar.sub` with `@[progress]` specs |
| Scalar access | `.toNat` | `.val` (returns `Nat` for unsigned) |
| Proof friction | Needs workarounds documented in [HAX_MISSING_LEMMAS.md](https://github.com/runtimeverification/p3-hax-lean-fri-pipeline/blob/main/HAX_MISSING_LEMMAS.md) | Standard Lean tactics suffice |

**Key takeaway:** The `bounded_by_max_log_arity` proof required zero custom lemmas with Aeneas.
Standard `generalize`/`cases`/`by_cases`/`simp` + Aeneas's `bind_tc_ok`/`if_pos`/`if_neg` were sufficient.
