# Aeneas FRI Fold-Arity Verification

Proof-of-concept for verifying Plonky3 FRI cryptographic code using
**Aeneas** (MIR-based, pure functional extraction) as an alternative to
**Hax** (annotation-driven, monadic extraction).

Reference Hax project: [p3-hax-lean-fri-pipeline](https://github.com/runtimeverification/p3-hax-lean-fri-pipeline)

## Pipeline

```
Rust source  ──Charon──▸  LLBC  ──Aeneas──▸  Lean 4  ──proofs──▸  verified properties
fold_arity/src/lib.rs    .llbc             FoldArity.lean        BasicProofs.lean
```

## Hax vs Aeneas Comparison

| Dimension | Hax | Aeneas |
|-----------|-----|--------|
| Extraction input | Rust HIR + annotations | Rust MIR (via Charon LLBC) |
| Monad | `RustM` (custom inductive) | `Result` (ok / fail / div) |
| Option type | `Core_models.Option.Option` (reversed ctors) | Standard Lean `Option` |
| Conditionals | `Bool.casesOn` — `split_ifs` silently fails | Standard `if/then/else` — `split_ifs` works |
| Checked arithmetic | Requires custom `-?` lemmas | Built-in via `UScalar.sub` with `@[progress]` specs |
| Scalar access | `.toNat` | `.val` (returns `Nat` for unsigned) |
| Proof friction | Needs workarounds documented in [HAX_MISSING_LEMMAS.md](https://github.com/runtimeverification/p3-hax-lean-fri-pipeline/blob/main/HAX_MISSING_LEMMAS.md) | Standard Lean tactics suffice |

**Key takeaway:** The `bounded_by_max_log_arity` proof required zero custom lemmas with Aeneas.
Standard `generalize`/`cases`/`by_cases`/`simp` + Aeneas's `bind_tc_ok`/`if_pos`/`if_neg` were sufficient.

### FoldingCorrectness.lean — Theorem Inventory

Connects Aeneas-extracted code to [ArkLib](https://github.com/Verified-zkEVM/ArkLib)
FRI specifications (`SingleRound`, `SplitFold`, `Domain`).

| # | Theorem | Status | Property |
|---|---------|--------|----------|
| 1 | `compute_log_arity_gives_folding_degree` | Proved | result maps to folding degree |
| 2 | `folding_respects_final_height` | Stated | domain height &ge; final after fold |
| 3 | `folding_reduces_degree` | Stated | `foldNth` reduces poly degree by arity |
| 4 | `round_consistency_preserved` | Stated | consecutive rounds &le; total distance |
| 5 | `arity_determines_domain_step` | Stated | subdomain containment via `evalDomain` |
| &mdash; | 2 concrete examples | Proved | `log_to_arity 7 = 128`, height arithmetic |

> **Note:** Aeneas and ArkLib (via Batteries/mathlib) both define
> `BitVec.toNat_pow`, so they cannot be imported in the same Lean environment.
> `FoldingCorrectness.lean` imports only ArkLib and axiomatizes the Aeneas-side
> properties (proved in `BasicProofs.lean`). Both are built as separate
> `@[default_target]` entries in `lakefile.lean`.

### Full FRI verification

This project verifies `compute_log_arity_for_round` — the function that determines folding arity at each FRI round. The proofs in `BasicProofs.lean` establish correctness properties for this function using concrete examples and bounded-output guarantees.

The project also includes `FoldingCorrectness.lean` and an `ArkLib` dependency, which prepare the foundation for the next stage of verification: proving correctness of functions that perform polynomial folding. This includes operations such as `fold_even_odd` and full FRI prover/verifier rounds that invoke foldNth on evaluation domain elements. At that stage, `ArkLib's` `SingleRound`, `SplitFold`, and `Domain` modules will provide the algebraic framework needed to connect the extracted Rust semantics to the underlying mathematical properties of FRI.
